ISSUE 2 — OrchestratorConfig must be defined explicitly in the schema
schema.txt references OrchestratorConfig as the second input to the loop
but specifies none of its fields. This is a structural gap.
Required fields to define in schema.txt:
  max_iterations: int (default 3)
  poll_interval_seconds: int (default 15)
  max_poll_attempts: int (default 40)
  log_root: Path or None (None triggers fallback)
  autonomous: bool (default False — feature flag, False = current single-shot)
  ollama_enabled: bool (default False)
  ollama_base_url: str
  ollama_model: str

ISSUE 3 — Poll loop vs HTTP connection lifetime must be committed to
schema.txt line 68 implies a synchronous poll loop inside run_in_threadpool
but local preproc can run for tens of minutes. This is architecturally
incompatible at that timescale without an explicit commitment.
Two paths, Codex must choose one:

  Path A (recommended given existing architecture):
  Loop is synchronous, capped at max_poll_attempts * poll_interval_seconds.
  Returns timed_out with job_id if cap reached. Caller polls
  /api/local-train/jobs/{job_id} for final state.

  Path B (correct long-term):
  Orchestrator loop runs as detached background worker. API returns accepted
  immediately with job_id. Caller polls for terminal state. Requires a new
  background task pattern and status endpoint.

FRAMEWORK RECOMMENDATION (new — for Codex to accept or rebut)
--------------------------------------------------------------
The assess/act/observe/repeat loop described in schema.txt is the ReAct pattern
(Yao et al. 2022). The recommended framework to implement it is LangGraph.

Reasons specific to this design:
- Explicit state machine: graph nodes map directly to assess/act/observe steps.
  Control flow is visible and enforced by structure, not by a while loop.
- Conditional edges express the routing logic (ready->train, preproc_needed->preproc,
  blocked->terminal) without nested if/else in the orchestrator function.
- Loop termination (max_iterations, terminal state) is first-class, not ad hoc.
- Tool-use is native. Matches the tool interface contract in schema.txt exactly.
- Runs synchronously in run_in_threadpool (Path A) or as async background task
  (Path B) without changing the graph definition.
- State persistence via checkpointer: timed_out jobs can be resumed later as a
  config change, not a rewrite.
- When ollama_enabled=False, graph uses deterministic edges and calls tools as
  plain Python functions. When ollama_enabled=True, Ollama drives the graph via
  tool calls. No structural change required for the switchover.

MCP LAYER (new — for Codex to accept or rebut)
-----------------------------------------------
mcp_server.py already exists in the control plane. The tool interface contract
in schema.txt (typed inputs, typed outputs, side-effect isolated) is MCP-shaped.

Recommended architecture:
  FastAPI (web.py)
      -> Orchestrator loop (LangGraph)
          -> Tools via MCP server (mcp_server.py exposes control-plane functions)
              -> fran / local_registry / hpc

MCP and LangGraph answer different questions and are not alternatives:
- MCP is the tool exposure protocol (how the LLM discovers and calls tools).
- LangGraph is the orchestration framework (when to call which tool and what to
  do with the result).

When ollama_enabled=False: LangGraph calls tools as direct Python imports.
When ollama_enabled=True: Ollama client receives MCP tool manifest and drives
the graph via tool calls. Same tool implementations, two calling conventions.
No structural change required for the switchover.

This design is forward-compatible: mcp_server.py already exists, tools are
already side-effect isolated, and the MCP layer requires no changes to fran,
local_registry, or hpc.
