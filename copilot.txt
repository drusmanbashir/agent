NEW — NATURAL LANGUAGE RESOLUTION CHAIN
-----------------------------------------
This section defines how free-text user input resolves to fran CLI execution
in the Ollama + LangGraph + MCP architecture. Codex must incorporate this
into the schema as it defines the full intent-to-execution contract.

Example input: "init project kitabd ds kits23"

Step 1 — NL to structured tool call (Ollama)
The LLM receives the user input alongside a system prompt that enumerates
available tools with typed signatures and argument descriptions. Example:

  Tool: init_project
  Description: Initialise a FRAN project and register its datasources.
  Arguments:
    title: str — project title, e.g. kitabd
    datasources: list[str] — registered dataset keys, e.g. ["kits23", "lits"]

The model maps "init project" -> init_project and "ds kits23" ->
datasources=["kits23"] from the tool description alone. No intent vocabulary
table exists. The tool schema IS the vocabulary. The model emits:

  {"tool": "init_project", "arguments": {"title": "kitabd", "datasources": ["kits23"]}}

Step 2 — Tool call to Python function (LangGraph)
LangGraph receives the structured tool call and routes it to the registered
Python function:

  @tool
  def init_project(title: str, datasources: list[str]) -> StatusResult:
      return service.project_ready(title=title, datasources=datasources, ...)

Step 3 — Python function to fran execution (existing code, unchanged)
service.project_ready -> ensure_project_local -> fran.run.project.project_init.main
fran reads datasets.yaml to resolve "kits23" to its folder path.
This layer is entirely existing code. No changes required.

Full chain:
  "init project kitabd ds kits23"
    -> Ollama: NL -> tool call JSON
    -> LangGraph: routes to registered Python function
    -> service.project_ready
    -> fran.run.project.project_init.main
    -> datasets.yaml -> kits23 path -> project initialised

ARCHITECTURAL IMPLICATIONS FOR SCHEMA
---------------------------------------
1. The tool schema description (name, argument names, argument descriptions,
   examples) is the vocabulary that governs NL resolution accuracy.
   Vague descriptions produce mapping errors. Each tool registered in
   mcp_server.py must have:
     - a precise one-line description of what it does
     - typed arguments with descriptions that include valid example values
     - explicit note of what keys are valid (e.g. dataset keys must appear
       in datasets.yaml — list them or reference where to find them)

2. This means mcp_server.py is not just a protocol adapter. It is the
   authoritative tool manifest that the LLM uses to resolve intent.
   It must be kept in sync with the actual tool signatures in service.py.
   Drift between mcp_server.py descriptions and service.py signatures is
   the primary failure mode for NL resolution.

3. The system prompt must constrain output format to JSON tool calls.
   Without this, Ollama returns prose and the parser breaks. This is a
   required field in OrchestratorConfig (system_prompt: str or a default
   template) and must be defined in the schema.

4. When ollama_enabled=False, Steps 1 is bypassed entirely. The
   TrainRequest Pydantic model serves as the structured input directly,
   and LangGraph calls tools via deterministic edges. The resolution chain
   collapses to: validated TrainRequest -> LangGraph -> tools -> fran.
   This is why TrainRequest must not be in models.py — it plays the role
   of the structured tool call output in the deterministic path.
