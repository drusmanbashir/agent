NEW — HPC INTEGRATION DESIGN
------------------------------
The existing agent/hpc/ workflow already implements:
- hpc_submit_poll_fetch.sh: submit + detached poll worker + job.meta provenance
- hpc_poll_worker.sh: per-job detached worker, lock/sentinel aware, no duplicates
- hpc_poll_logs.sh: canonical log fetch, stamps last_polled_at in registry
- job_registry.py: Python abstraction over job_registry.tsv, derives active/closed,
  resolves per-job local paths (job_dir, std.out, std.err, worker logs)
- cli/job_registry.sh: add/submit/finish/show/find/ids/archive operations on TSV
- hpc_dashboard_web.py: web dashboard, active/closed tabs, poll actions, log views
- cli/train.sh: Slurm GPU training payload (train_retry.py)
- cli/preproc.sh: Slurm preproc payload (analyze_resample.py)
- queue_project_init_preproc.sh: paired project-init + dependent preproc Slurm jobs
- hpc_resubmit.sh: replay prior submit from stored job.meta

These are mature, working implementations. The orchestration refactor must not
duplicate them. The integration principle is:

  hpc/ stays as the execution and polling layer.
  The orchestrator calls into hpc/ tools as its submit/observe implementations
  when mode=hpc, exactly as it calls local_registry for mode=local.
  No new polling logic. No new job identity logic. No new registry logic.

HPC INTEGRATION BOUNDARIES
----------------------------

1. submit_preproc tool (hpc mode)
   Calls cli/preproc.sh via hpc_submit_poll_fetch.sh through hpc.py.
   Returns job_id, job_dir, poll_url, stdout_path, stderr_path in StatusResult.
   job_id comes from hpc_submit_poll_fetch.sh output. No new submit mechanism.

2. submit_train tool (hpc mode)
   Calls cli/train.sh via hpc_submit_poll_fetch.sh through hpc.py.
   Same return contract as local: job_id, job_dir, poll_url, log paths.
   hpc_submit_poll_fetch.sh already writes job.meta with submit_argv for resubmit.

3. observe_prereq / observe_train_short tools (hpc mode)
   Calls hpc_poll_logs.sh for a single poll cycle via hpc.py.
   Reads result from job_registry.tsv via job_registry.py.
   Returns current state (submitted, running, completed, failed, cancelled).
   No new polling worker. The detached hpc_poll_worker.sh already runs in background.
   The orchestrator observe step is a registry READ, not a new poll mechanism.

4. build_crash_packet tool (hpc mode)
   Reads std.out / std.err from job_dir (already fetched by hpc_poll_worker.sh).
   Reads worker.log, poll.log from job_dir (already written by hpc_poll_worker.sh).
   Reads job.meta for provenance (already written at submit time).
   Reads job_registry.tsv for terminal state / exit_code.
   No new data collection. Crash packet assembly reads what already exists.

5. rerun_failed_stage tool (hpc mode)
   Calls hpc_resubmit.sh <job_id>, which replays submit_argv from job.meta.
   Returns new job_id. No new resubmit mechanism needed.

BOUNDARY CHANGES REQUIRED IN hpc.py
--------------------------------------
hpc.py currently exposes submit and poll as direct function calls. Two changes:

A. All hpc tool functions must return StatusResult, not raw dicts or shell output.
   hpc.py wraps the existing CLI calls and maps their output to StatusResult.
   The orchestrator never reads raw TSV rows or shell stdout directly.

B. observe functions must accept job_id as input and return StatusResult.
   Currently hpc_poll_logs.sh resolves "last" from the registry. The orchestrator
   needs to poll a specific job_id from its graph state, not the last-seen job.
   hpc_poll_logs.sh already accepts <job_id> as an argument. No shell change needed.
   hpc.py must pass job_id explicitly, not rely on "last" resolution.

WEB INTERFACE — REQUIRED ADAPTATIONS
---------------------------------------
The existing web UI (web.py + Jinja2 templates) is a human-in-the-loop form flow.
It is correct for its purpose. Minimal changes only:

1. The train submission form currently POSTs fields individually via Body().
   Replace with TrainRequest Pydantic model. Form field names stay the same.
   No visible UI change. Only server-side validation improves.

2. Add a mode selector to the train form: local | hpc.
   This sets OrchestratorConfig.mode and routes submit/observe to the correct
   tool implementations. Currently mode is implicit/hardcoded.

3. Status display must handle the new terminal states: submitted, timed_out,
   debug_failed, repair_submitted. The existing breakpoint display can be removed
   since the orchestrator no longer returns breakpoints to the HTTP client.

4. The HPC dashboard (hpc_dashboard_web.py) is a separate service and stays
   separate. It is not merged into the FastAPI web.py. Cross-linking is sufficient:
   StatusResult.details.dashboard_url already points to the dashboard for a job.
   The web UI can render this as a link. No architectural merge needed.

5. The timed_out response includes poll_url. The UI should render this as a
   direct link so the user can monitor the background job without a new form submit.
