Boundary
Control-plane path: /home/ub/code/agent/agent/control_plane local training orchestration, from FastAPI request intake through readiness assessment, local/HPC preprocessing submission, local train submission, registry/log observation, and terminal response. Data-plane execution remains FRAN preprocess/train scripts and HPC/local workers. Dependency edges are /home/ub/code/fran entrypoints, /home/ub/code/agent/agent/hpc registry tooling, /s/fran_storage/conf, /s/agent_rw/hpc_logs, and the future Ollama endpoint if retained.

Current defects
1. The orchestrator is not at the top of the call stack. /api/orchestrator/requests/train calls service readiness and a small decision helper, then either returns a breakpoint or submits training.
2. Preprocessing is gated by client follow-up. train_plan_ready returns breakpoint=preproc instead of letting the orchestrator submit preproc, observe completion, and continue.
3. provider/model are recorded but not used as a real client config. They are misleading operational metadata unless wired to Ollama for decisions.
4. Status payloads are not canonical. Status strings and fields vary across train/preproc/job paths; blocked is used ad hoc but absent from models.STATUSES.
5. The train API uses per-field Body parameters, so validation and versioning are spread across web.py and service signatures.
6. Local logs assume /s/agent_rw/hpc_logs exists. A developer or fallback runtime without /s mounted cannot run the local registry path.

Target architecture
The orchestrator owns the train intent lifecycle and sits above all tool access:
request -> validate TrainRequest -> orchestrator loop -> assess -> act -> observe -> repeat -> terminal StatusResult.

/api/orchestrator/requests/train remains the public endpoint, but semantics change from gated single-shot to an autonomous agent loop returning terminal state. Terminal means submitted train, completed train if configured to wait, blocked, failed, or timed out. It must not return breakpoints that require the caller to manually invoke preproc.

Provider/model decision
Choose one path before rollout:
1. Real config: add an Ollama client config with base_url, model, timeout, and enabled flag, and use it only inside the orchestrator decision boundary.
2. Remove fields: delete provider/model from request, local registry metadata, and orchestrator state until there is a real client.
Preferred smallest safe path: remove provider/model from behavior-facing contracts unless an Ollama endpoint is available and validated.

Execution loop contract
Input is a validated TrainRequest and an OrchestratorConfig.
Each iteration records an observation with status, message, details, and observed_at.
Assess determines current readiness from tools, not cached assumptions.
Act may submit preproc or train exactly once per missing step unless observation proves retry is needed.
Observe polls the submitted artifact through registry/HPC/local tools until ready, failed, blocked, or timeout.
Repeat continues only while progress is possible and within configured max_iterations or deadline.
Return one terminal StatusResult. No breakpoint field. Use next_action only for operator-visible guidance after terminal blocked/failed states.

Tool interface contract
Tools are the only side-effect boundary below the orchestrator.
Each tool accepts typed inputs and returns StatusResult.
Tools must be idempotent where practical: repeated assess is read-only, repeated submit checks existing active work before creating a new job, repeated observe does not mutate except registry poll timestamps.
Tools must surface job_id, command, job_dir, dashboard_url, and log_root in details/job when applicable.
Tools do not call the orchestrator and do not decide high-level policy.

Request/response contract
Request: POST /api/orchestrator/requests/train with JSON matching TrainRequest.
Response: canonical StatusResult dict.
Allowed terminal statuses: submitted, completed, blocked, failed, timed_out if timeout is added to STATUSES.
submitted means train was accepted by the local/HPC execution substrate.
blocked means the orchestrator cannot safely repair the prerequisite without operator action.
failed means an attempted action failed.
details must include plan_id, mode, dashboard, log_root, and last_observation when available.

Validation
Normal path: project plan already has source and plan datasets; endpoint returns submitted train with canonical StatusResult and job/log details.
Failure path: project is missing or preproc/train submission fails; endpoint returns blocked or failed with error_code, next_action, and no partial schema.
Recovery path: /s/agent_rw/hpc_logs unavailable; local submission writes registry and artifacts under ~/.agent/hpc_logs and subsequent job detail/crash endpoints resolve the same job.
Autonomous path: plan needs preprocessing; orchestrator submits preproc, observes completion, reassesses readiness, then submits train without returning a breakpoint.
Determinism: repeated requests with active matching preproc/train work observe existing work instead of duplicating submissions.

Rollback
Keep route name stable so rollback is code-level only.
Feature-flag the autonomous loop during rollout; disabling it restores previous readiness-submit behavior.
Keep old response keys only as temporary compatibility inside details, not as top-level contract.
If log fallback causes registry ambiguity, force configured log root back to /s/agent_rw/hpc_logs and stop local fallback.
If Ollama config is not validated, remove provider/model rather than retaining inert fields.
