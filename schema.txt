Boundary

Data-plane execution remains FRAN preprocess/train scripts and HPC/local workers. Current defects
1. The orchestrator is not at the top of the call stack. /api/orchestrator/requests/train calls service readiness and a small decision helper, then either returns a breakpoint or submits training.
2. Preprocessing is gated by client follow-up. train_plan_ready returns breakpoint=preproc instead of letting the orchestrator submit preproc, observe within a cap, and continue when possible.
3. provider/model are recorded but not used as a real client config. They must become concrete Ollama config or be removed.
4. Status payloads are not canonical. Status strings and fields vary across train/preproc/job paths; blocked is used ad hoc but absent from models.STATUSES.
5. The train API uses per-field Body parameters, spreading request validation across web.py and service signatures.
6. Local logs assume /s/agent_rw/hpc_logs exists. A developer or fallback runtime without /s mounted cannot reliably use the local registry path.

Target architecture
The orchestrator owns the train intent lifecycle and sits above all tool access:
HTTP request -> validate web request schema -> build domain command/config -> orchestrator loop -> assess -> act -> observe -> repeat -> terminal StatusResult.

/api/orchestrator/requests/train remains the public endpoint. Its semantics change from gated single-shot to a capped autonomous agent loop. The endpoint returns one terminal StatusResult for the HTTP attempt: submitted, completed, blocked, failed, or timed_out. It must not return breakpoints that require the caller to manually invoke preproc.

HTTP lifetime commitment: Path A
The autonomous loop runs synchronously inside run_in_threadpool and is capped by OrchestratorConfig.max_iterations, max_poll_attempts, poll_interval_seconds, and deadline_seconds.
If the cap is reached while preproc or train work is still active, return status=timed_out with active job identity, log_root, and poll_url in details. The caller then polls existing job endpoints such as /api/local-train/jobs/{job_id}, /api/local-train/jobs/{job_id}/crash, stdout, and stderr.
timed_out is terminal for the HTTP request, not proof that the underlying job failed. The orchestrator must persist enough registry/provenance state for later polling to recover the live state.

Provider/model decision
Choose one path before implementation:
1. Real config: keep Ollama only through OrchestratorConfig fields ollama_enabled, ollama_base_url, ollama_model, and ollama_timeout_seconds. Use it only inside the orchestrator decision boundary.
2. Remove fields: delete provider/model from request, local registry metadata, and orchestrator state until there is a real client.
Preferred smallest safe path: remove provider/model from behavior-facing contracts unless an Ollama endpoint is available and validated.

OrchestratorConfig
Define an explicit config object in the control-plane orchestration layer, not in web request schemas:
autonomous: bool, default true. False restores current single-shot readiness-submit behavior for rollback.
max_iterations: int, bounds total assess/act/observe cycles.
poll_interval_seconds: float, sleep between observe attempts.
max_poll_attempts: int, bounds observe polling for each submitted action.
deadline_seconds: float, hard cap for the synchronous HTTP attempt.
log_root: Path | None, where None means prefer /s/agent_rw/hpc_logs when writable and otherwise use ~/.agent/hpc_logs.
mode: str, local or hpc.
ollama_enabled: bool, default false unless real client is configured.
ollama_base_url: str | None.
ollama_model: str | None.
ollama_timeout_seconds: float.

Execution loop contract
Input is a validated train command derived from TrainRequest plus OrchestratorConfig.
Each iteration records an observation with status, message, details, and observed_at.
Assess determines current readiness from tools, not cached assumptions.
Act may submit preproc or train exactly once per missing step unless observation proves retry is needed.
Observe polls the submitted artifact through registry/HPC/local tools until ready, submitted, running, completed, blocked, failed, or timed_out.
Repeat continues only while progress is possible and within configured iteration, poll, and deadline caps.
Return one terminal StatusResult for the HTTP attempt. No breakpoint field. Use next_action only for operator-visible guidance after terminal blocked, failed, or timed_out states.

Tool interface contract
Tools are the only side-effect boundary below the orchestrator.
Each tool accepts typed inputs and returns StatusResult.
Tools must be idempotent where practical: repeated assess is read-only, repeated submit checks existing active work before creating a new job, repeated observe does not mutate except registry poll timestamps.
Tools must surface job_id, command, job_dir, dashboard_url, poll_url, and log_root in details/job when applicable.
Tools do not call the orchestrator and do not decide high-level policy.

Request/response contract
Request: POST /api/orchestrator/requests/train with JSON matching web-facing TrainRequest.
Response: canonical StatusResult dict.
Allowed terminal statuses for this endpoint: submitted, completed, blocked, failed, timed_out.
submitted means train was accepted by the local/HPC execution substrate within the HTTP cap.
completed means the configured cap allowed observation through successful completion.
blocked means the orchestrator cannot safely repair the prerequisite without operator action.
failed means an attempted action failed.
timed_out means the capped synchronous loop reached its limit while work may still be active; details must include poll_url and active job/log identity when available.
details must include plan_id, mode, dashboard, log_root, last_observation, and poll_url when available.

Validation
Normal path: project plan already has source and plan datasets; endpoint returns submitted train with canonical StatusResult and job/log/poll details.
Autonomous path: plan needs preprocessing; orchestrator submits preproc, observes within cap, reassesses readiness, then submits train if ready before deadline.
Timed-out path: long preproc or train exceeds cap; endpoint returns timed_out with active job_id, log_root, last_observation, and poll_url; existing job detail/crash/stdout/stderr endpoints can retrieve state.
Failure path: project is missing or preproc/train submission fails; endpoint returns blocked or failed with error_code, next_action, and no partial top-level schema.
Recovery path: /s/agent_rw/hpc_logs unavailable; local submission writes registry and artifacts under ~/.agent/hpc_logs and subsequent polling resolves the same job.
Determinism: repeated requests with active matching preproc/train work observe existing work instead of duplicating submissions.

Rollback
Keep route name stable so rollback is code-level only.
Set OrchestratorConfig.autonomous=false to restore previous readiness-submit behavior while keeping typed request and canonical status response.
Keep old response keys only as temporary compatibility inside details, not as top-level contract.
If timed_out polling semantics are unsafe in practice, lower caps and return submitted earlier with poll_url rather than holding HTTP longer.
If log fallback causes registry ambiguity, force configured log_root back to /s/agent_rw/hpc_logs and stop local fallback.
If Ollama config is not validated, remove provider/model rather than retaining inert fields.
